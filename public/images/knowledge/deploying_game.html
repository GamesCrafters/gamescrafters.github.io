Deploying the Game

Once your game is solved and plugged into the Gamesman ecosystem, the last step is to actually put it on the website!

This involves:

  • Starting the appropriate game server (Classic, One, or Puzzles).
  • Running the GamesCraftersUWAPI server that Uni talks to.
  • Running the GamesmanUni web app pointing at your local UWAPI.

After doing this, you should be able to open Uni in your browser, pick your game, and see your solver’s values show up through the AutoGUI.

Local Setup

A typical local setup looks like this:

  1. Game server (GamesmanClassic, GamesmanOne, or GamesmanPuzzles) exposes your game’s data.
  2. GamesCraftersUWAPI wraps those backends with a unified JSON API.
  3. GamesmanUni is the frontend that players actually see.

Running GamesmanClassic Locally

If your game is implemented in GamesmanClassic, you’ll want to run the following: 

# Clone GamesmanClassic
git clone https://github.com/GamesCrafters/GamesmanClassic.git
cd GamesmanClassic

# Configure and build (graphics disabled)
./configure --disable-graphics
make

# Install any Python dependencies for helper scripts
pip install -r requirements.txt

# Run the Classic server
python3 src/py/server.py

This starts the server on port 8083. Check the README in GamesmanClassic for any additional information.

Running GamesmanOne or GamesmanPuzzles Locally

GamesmanOne

For games written in the GamesmanOne framework: 

# Clone GamesmanOne
git clone https://github.com/GamesCrafters/GamesmanOne.git
cd GamesmanOne

# Recompile your code and start the server
bash ./gamesman_install.sh  # and follow interactive instructions
python3 server.py

This starts the server on port 8084.

GamesmanPuzzles

For puzzles implemented in GamesmanPuzzles: 

# Clone GamesmanPuzzles
git clone https://github.com/GamesCrafters/GamesmanPuzzles.git
cd GamesmanPuzzles

pip install -r requirements.txt
pip install -e .
mkdir databases

# Run the Puzzles server
python3 -m scripts.server

This starts the server on port 9001.

Running GamesCraftersUWAPI Locally

GamesCraftersUWAPI is the middleware layer that speaks a uniform API to Uni and forwards requests to Classic / One / Puzzles. 

# Clone GamesCraftersUWAPI
git clone https://github.com/GamesCrafters/GamesCraftersUWAPI.git
cd GamesCraftersUWAPI

pip install -r requirements.txt

You can sanity-check it with URLs like:

  • http://127.0.0.1:8082/games/
  • http://127.0.0.1:8082/<game>/<size>/instructions/
  • http://127.0.0.1:8082/<game>/<size>/positions/?p=...

Running GamesmanUni Locally

Finally, you need GamesmanUni, the frontend web app. This is what players actually use in their browser. 

# Clone GamesmanUni
git clone https://github.com/GamesCrafters/GamesmanUni.git
cd GamesmanUni

# Install JS dependencies
yarn        # or: npm install

# Start a dev server (if available)
yarn dev    # or: npm run dev

Uni needs to know where your UWAPI server lives. In older versions this is configured in a small config or settings file, often something like: 

const config = {
  uniAPI:  "http://localhost:8082/universal/v1/",
  gameAPI: "http://localhost:8082/"
};

Once configured, open the URL printed by yarn dev (often http://localhost:5173 or similar) and try selecting your game.

Testing Your Local Deployment

A good checklist:

  1. Hit your game server directly (Classic / One / Puzzles) with a simple curl/browser request.
  2. Hit UWAPI and confirm it can see your game and return position data.
  3. Run Uni and confirm that:
    • Your game appears in the game list.
    • Moves update the board correctly.
    • Values / remotenesses / winbys show up as expected.
🧪
If something is broken, check from the bottom up: game server → UWAPI → Uni. That makes it easier to see where data stops flowing.

Deploying to the NYC Servers (Staff Only)

If you’re a GamesCrafters lead or staff member, you may also deploy your changes to the shared NYC servers so others can use them. The high-level pattern is:

  1. SSH into nyc.cs.berkeley.edu as gamescrafters.
  2. Pull the latest code in the relevant repo.
  3. Rebuild if needed (for C/C++ projects like Classic).
  4. Restart the appropriate systemd service.
  5. For Uni, build the frontend and rsync it into place.

Example: Restarting Puzzles

ssh gamescrafters@nyc.cs.berkeley.edu
cd ~/GamesmanPuzzles
git pull
sudo systemctl restart gmp

Example: Updating Uni

ssh gamescrafters@nyc.cs.berkeley.edu
cd ~/GamesmanUni
yarn
yarn build
sudo rsync -a --delete ./dist/ /var/www/html/uni/

Exact commands and service names can change; always check current internal documentation or ask a lead before touching production.

Wrapping Up

You now know how to bring your game all the way from a solver running in isolation to a full Uni + UWAPI + backend stack other people can play with.

  • Run the backend your game lives in (Classic, One, or Puzzles).
  • Run GamesCraftersUWAPI pointing at that backend.
  • Run GamesmanUni pointing at your UWAPI.
  • Optionally, deploy to the shared NYC servers if you’re staff.

In practice, you’ll repeat this loop many times as you debug, tune, and improve your game.
