Tic-Tac-Toe: A Complete Walkthrough

This page walks you through how Tic-Tac-Toe is actually implemented in the GamesCrafters system, from game logic and hashing, to AutoGUI, to running the full local stack and seeing it live in GamesmanUni.

💡
This page assumes you've read the earlier chapters on game trees, position values, hashing, and AutoGUI. Links to those chapters are in the sidebar if you need a refresher.

1. The Game at a Glance

Tic-Tac-Toe is played on a 3×3 grid. Two players alternate placing their mark (X goes first, then O). The first player to get three of their marks in a row—horizontally, vertically, or diagonally—wins. If all nine squares are filled with no winner, it's a draw.

The user-facing legend numbers cells 1–9 (top-left to bottom-right). Internally, the implementation uses indices 0–8—exactly one less than what the player types. The conversion happens in ConvertTextInputToMove: input[0] - '1'.

Key facts for the solver:

  • Board size: 9 cells, each one of {Blank, o, x}.
  • Maximum game length: 9 moves.
  • Total hash space: 19,683 (= 39), of which 5,478 are reachable positions.
  • Possible outcomes: Win, Lose, or Tie.
  • Tic-Tac-Toe is a draw with perfect play from both sides.

2. Representing a Position

Unlike many modern GamesCrafters implementations that use string-based positions, GamesmanClassic encodes the board as a single integer in base 3. Each cell contributes one base-3 digit according to the BlankOX enum: 

typedef enum possibleBoardPieces {
    Blank = 0,
    o     = 1,
    x     = 2
} BlankOX;

The position integer is the sum of each cell's value times the corresponding power of 3: 

/* Powers of 3: the encoding weights for each cell */
int g3Array[] = { 1, 3, 9, 27, 81, 243, 729, 2187, 6561 };

/* Encoding: position = sum of (piece_value * 3^cell_index) for all cells */
POSITION BlankOXToPosition(BlankOX *theBlankOX) {
    int i;
    POSITION position = 0;
    for (i = 0; i < BOARDSIZE; i++)
        position += g3Array[i] * (int)theBlankOX[i];
    return position;
}

The all-blank starting board encodes to position 0 (gInitialPosition = 0). The maximum possible value is 2×(1+3+9+…+6561) = 2×9841 = 19,682, giving a total hash space of 39 = 19,683 slots (positions 0–19,682).

The inverse—decoding an integer back to a board—peels off digits from high to low: 

void PositionToBlankOX(POSITION thePos, BlankOX *theBlankOX) {
    int i;
    for (i = 8; i >= 0; i--) {
        if      (thePos >= (POSITION)(x * g3Array[i]))
            { theBlankOX[i] = x;     thePos -= x * g3Array[i]; }
        else if (thePos >= (POSITION)(o * g3Array[i]))
            { theBlankOX[i] = o;     thePos -= o * g3Array[i]; }
        else
            { theBlankOX[i] = Blank; thePos -= Blank * g3Array[i]; }
    }
}

Whose turn it is is not stored in the position—it's derived on the fly by counting pieces. X always goes first, so equal counts mean it's X's turn: 

BlankOX WhoseTurn(BlankOX *theBlankOX) {
    int i, xcount = 0, ocount = 0;
    for (i = 0; i < BOARDSIZE; i++)
        if      (theBlankOX[i] == x) xcount++;
        else if (theBlankOX[i] == o) ocount++;
    return (xcount == ocount) ? x : o;
}
💡
Encoding pieces as 0/1/2 and using powers of 3 is a classic trick: it makes DoMove a single arithmetic addition and keeps the entire board state in one integer with no struct overhead.

3. Hashing Positions

Because the position is the hash, there's nothing extra to implement. The base-3 encoding described in the previous section gives every board a unique integer in [0, 19683), and the solver allocates an array of that size directly: 

POSITION gNumberOfPositions = 19683;  /* 3^9 */

Most of those slots are never reached during a real game—only 5,478 positions are actually reachable from the starting board. The solver simply ignores the unreachable slots; they remain Undecided after solving.

📊
From the source file comments, the solver outcome breakdown for the 5,478 reachable positions is: Win: 2,836  ·  Lose: 1,574  ·  Tie: 1,068. With symmetry reduction (SLIMFAST), this shrinks to just 3,481 canonical positions.

The tradeoff versus a tighter combinatorial hash is deliberate: the base-3 scheme wastes some space (19,683 allocated vs. 5,478 used) but makes DoMove and Primitive extremely fast—no hash/unhash step needed in the hot path.

4. Primitive Check and Move Generation

Checking for a win: ThreeInARow and Primitive

The helper ThreeInARow checks three specific cells for a matching, non-blank piece: 

BOOLEAN ThreeInARow(BlankOX *theBlankOX, int a, int b, int c) {
    return (theBlankOX[a] == theBlankOX[b] &&
            theBlankOX[b] == theBlankOX[c] &&
            theBlankOX[c] != Blank);
}

Primitive checks all 8 lines (3 rows, 3 columns, 2 diagonals). If any line is complete, the current player (the one who must move next) has just lost—because the previous player made the winning move. The board-full case is a tie: 

VALUE Primitive(POSITION position) {
    PositionToBlankOX(position, gPosition.board);

    if (ThreeInARow(gPosition.board, 0, 1, 2) ||
        ThreeInARow(gPosition.board, 3, 4, 5) ||
        ThreeInARow(gPosition.board, 6, 7, 8) ||
        ThreeInARow(gPosition.board, 0, 3, 6) ||
        ThreeInARow(gPosition.board, 1, 4, 7) ||
        ThreeInARow(gPosition.board, 2, 5, 8) ||
        ThreeInARow(gPosition.board, 0, 4, 8) ||
        ThreeInARow(gPosition.board, 2, 4, 6))
        return gStandardGame ? lose : win;   /* loser faces a finished board */
    else if (AllFilledIn(gPosition.board))
        return tie;
    else
        return undecided;
}
⚠️
A common mistake is returning win when three-in-a-row is found. But the solver convention is: the position's value is the value for the player who must move from it. If you're facing a board where the opponent already has three-in-a-row, you've already lost—so the value is lose. The gStandardGame flag flips this for misère variants.

Generating moves: GenerateMoves

A move is simply the index (0–8) of any empty cell: 

MOVELIST *GenerateMoves(POSITION position) {
    int index;
    MOVELIST *moves = NULL;
    PositionToBlankOX(position, gPosition.board);

    for (index = 0; index < BOARDSIZE; ++index)
        if (gPosition.board[index] == Blank)
            moves = CreateMovelistNode(index, moves);

    return moves;
}

Applying a move: DoMove

Because the position is a base-3 integer, applying a move is a single addition—no string manipulation needed. The piece value (1 for o, 2 for x) is multiplied by the cell's power of 3 and added to the position: 

POSITION DoMove(POSITION position, MOVE move) {
    BlankOX board[BOARDSIZE];
    PositionToBlankOX(position, board);
    /* WhoseTurn returns 'x' (=2) or 'o' (=1), which doubles as the piece value */
    return position + g3Array[move] * (int)WhoseTurn(board);
}

5. Symmetries and the Solver

GamesmanClassic's retrograde solver handles the game tree automatically once Primitive, GenerateMoves, and DoMove are implemented. Tic-Tac-Toe also supports symmetry reduction to drastically cut the number of positions that need to be stored.

The 8 board symmetries

A 3×3 board has 8 symmetries: 4 rotations (0°, 90°, 180°, 270°) and 4 flipped rotations. The implementation precomputes a matrix that maps each cell index to its new index under each symmetry, then selects the smallest integer among all 8 equivalent positions as the canonical representative: 

/* Rotate 90° clockwise: cell 0→6, 1→3, 2→0, ... */
int gRotate90CWNewPosition[] = { 6, 3, 0, 7, 4, 1, 8, 5, 2 };

/* Flip along N-S axis: cell 0→2, 1→1, 2→0, ... */
int gFlipNewPosition[] = { 2, 1, 0, 5, 4, 3, 8, 7, 6 };

POSITION GetCanonicalPosition(POSITION position) {
    POSITION theCanonical = position;
    for (int i = 0; i < NUMSYMMETRIES; i++) {
        POSITION newPos = DoSymmetry(position, i);
        if (newPos < theCanonical)
            theCanonical = newPos;
    }
    return theCanonical;
}

With full symmetry reduction ("SLIMFAST"), the 5,478 reachable positions collapse to just 3,481 canonical ones—a 1.57× space saving with no loss in solver correctness.

🔍
The solver propagates values bottom-up: primitive positions first, then their parents. With perfect play, Tic-Tac-Toe always ends in a tie. Remoteness varies by line of play and tells you how many moves until the inevitable outcome.

6. Wiring Up AutoGUI

Two C functions bridge the GamesmanClassic solver to AutoGUI: PositionToAutoGUIString and MoveToAutoGUIString. These are the functions you implement in every Classic game file to get a working visual frontend.

PositionToAutoGUIString

This function converts the internal base-3 integer into an AutoGUI position string. It first decodes the integer to a BlankOX array, converts each cell to its character ('x', 'o', or '-'), then calls the framework helper AutoGUIMakePositionString with the turn number (1 for X's turn, 2 for O's turn): 

void PositionToAutoGUIString(POSITION position, char *autoguiPositionStringBuffer) {
    BlankOX oxboard[BOARDSIZE];
    PositionToBlankOX(position, oxboard);

    char board[BOARDSIZE + 1];
    for (int i = 0; i < BOARDSIZE; i++) {
        if      (oxboard[i] == o) board[i] = 'o';
        else if (oxboard[i] == x) board[i] = 'x';
        else                      board[i] = '-';
    }
    board[BOARDSIZE] = '\0';

    AutoGUIMakePositionString(
        WhoseTurn(oxboard) == x ? 1 : 2,   /* turn: 1=X, 2=O */
        board,
        autoguiPositionStringBuffer
    );
}

For the example board xo--x-o-x with X to move, this produces the AutoGUI position string "1_xo--x-o-x". The leading 1_ tells AutoGUI it's player 1's (X's) turn.

MoveToAutoGUIString

This function produces an A-type move button string. Crucially, the token character is the current player's piece—'x' or 'o'—so AutoGUI draws the appropriate piece image as the move button, not a generic circle: 

void MoveToAutoGUIString(POSITION position, MOVE move, char *autoguiMoveStringBuffer) {
    BlankOX oxboard[BOARDSIZE];
    PositionToBlankOX(position, oxboard);
    char token = (WhoseTurn(oxboard) == x) ? 'x' : 'o';
    AutoGUIMakeMoveButtonStringA(token, move, '-', autoguiMoveStringBuffer);
}

The arguments to AutoGUIMakeMoveButtonStringA are:

  • token: the charImages key to use for the button ('x' or 'o')
  • move: the center index (0–8) where the button appears
  • '-': the sound key ('-' = no sound / default)

So for X placing at cell 4, the resulting move string is "A_x_4_-".

8. Registering the Game in UWAPI

Since mttt.c lives in GamesmanClassic, UWAPI routes requests through the Classic backend. The game is identified by its kDBName, which is "ttt" (not "tictactoe").

In UWAPI's Classic game registry, ensure an entry exists for "ttt" pointing at the running GamesmanClassic server: 

# In GamesCraftersUWAPI (Classic game routing config)
CLASSIC_GAMES = {
    # ... existing entries ...
    "ttt": {
        "name": "Tic-Tac-Toe",
        "variants": { "regular": {} }
    }
}

After registering, UWAPI will expose the game at: 

http://localhost:8082/classic/ttt/regular/positions/?p=0
📋
The initial position for Classic games is passed as the integer hash—p=0 for the all-blank starting board. UWAPI translates this to and from StringToPosition / PositionToAutoGUIString on the Classic side.

9. Running the Local Stack

Since Tic-Tac-Toe is a GamesmanClassic game, the full stack is: GamesmanClassic → GamesCraftersUWAPI → GamesmanUni.

4
Open Uni in your browser, find Tic-Tac-Toe in the game list, and click through a game. You should see:
  • The board SVG rendered as the background.
  • X and O piece images appearing as you make moves.
  • Move buttons showing the current player's piece on each empty cell.
  • Win / tie / lose values and remotenesses shown by Uni.

10. Debugging Common Issues

Board SVG doesn't appear

Check that tictactoe/board.svg is in GamesmanUni/public/images/ and that the path in your AutoGUI JSON matches exactly (case-sensitive on Linux).

Pieces appear at wrong positions

Verify your centers array matches your space setting (300×300). A center at [50, 50] means 50px from the left and 50px from the top of the board image. Remember that center index 0 corresponds to the first character of the entity string—which in mttt.c is cell 0 (top-left).

Move buttons show circles instead of X/O piece images

This happens when the move token is '-' instead of the current player's piece. In MoveToAutoGUIString, the first argument to AutoGUIMakeMoveButtonStringA must be 'x' or 'o' (derived from WhoseTurn), not a literal dash.

UWAPI returns 404 for "ttt"

The game isn't registered. Double-check that the key in UWAPI's Classic game registry matches kDBName = "ttt" from mttt.c exactly. Also confirm GamesmanClassic compiled successfully and its server is running before UWAPI starts.

Position values look wrong (all "lose" or all "win")

The primitive check is likely returning the wrong value. In GamesmanClassic convention, Primitive returns lose when three-in-a-row exists—because the current player (who must move) faces a board where the opponent already won. Never return win from Primitive in a standard game.

Turn indicator is wrong in Uni (shows O when it should be X)

Check the turn argument to AutoGUIMakePositionString. It must be 1 for X's turn and 2 for O's turn. The condition is WhoseTurn(oxboard) == x ? 1 : 2.

🧪
Debug from the bottom up: GamesmanClassic → UWAPI → Uni. If UWAPI returns correct JSON but Uni looks wrong, the issue is in the AutoGUI strings or JSON data. If UWAPI returns wrong values, check Primitive and the Classic server connection.

Wrapping Up

Here's what the actual mttt.c implementation provides, end to end:

  • A base-3 integer as the position representation (39 = 19,683 slots, 5,478 reachable).
  • The BlankOX enum {Blank=0, o=1, x=2} doubling as piece values.
  • DoMove as a single arithmetic addition: position + g3Array[move] * WhoseTurn.
  • Turn derived from piece counts via WhoseTurn—no extra flag stored.
  • Primitive checking 8 lines and returning lose (never win) on three-in-a-row.
  • 8-symmetry reduction via precomputed rotation/flip matrices, cutting canonical positions to 3,481.
  • PositionToAutoGUIString using turn 1 (X) or 2 (O) + the 9-char board string.
  • MoveToAutoGUIString with the current player's piece as the A-type button token.
  • AutoGUI JSON keyed as "ttt" (matching kDBName) with 9 centers on a 300×300 canvas.
  • The full Classic → UWAPI → Uni stack to bring it to players.

Once this is running you can experiment: enable or disable symmetry reduction via the options menu, try the misère variant (where three-in-a-row loses), or use mttt.c as a template for your own GamesmanClassic game.
